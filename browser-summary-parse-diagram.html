How the browser_summary parser works

The comment block is not displayed directly and it is not executed by Python. It is a tiny static spec that the runtime reads before the user's script runs. Later, every labeled emit_output call can use that spec to derive a compact transcript row.

short answer

One structured call, two consumers

emit_output(value, label="page_info") stores full data for the model. The label also selects a summary template for the frontend. No full page object has to be printed just to make the TUI readable.

1

Agent submits code

The final submitted Python source includes a top comment block plus ordinary script code.

input: source string
2

Runtime scans top lines

_parse_browser_summary_specs scans the first 80 lines, finds # browser_summary:, strips comment markers, and parses JSON.

output: label spec map
3

Script executes

Python comments do nothing at runtime. The actual event happens when script code calls emit_output.

input: value + label
4

Summary is derived

The helper keeps the full value, looks up the label, evaluates selectors like $.url, and appends both outputs and summary.

output: two channels
5

TUI renders summary

The model receives full structured output. The transcript prefers compact summary records and hides transport noise.

display: compact row

Submitted Python

The parser reads the comment block before execution. The label in the block must match the label in emit_output.

# browser_summary:
# {
#   "page_info": {
#     "kind": "page",
#     "url": "$.url",
#     "title": "$.title"
#   },
#   "employee_rows": {
#     "kind": "extracted",
#     "message": "Read ${$.length} employee rows"
#   }
# }

info = page_info()
emit_output(info, label="page_info")

rows = [{"name": "Ada"}, {"name": "Grace"}]
emit_output(rows, label="employee_rows")

Parsed Summary Spec

After stripping # prefixes, the runtime has a plain JSON object keyed by output label.

{
  "page_info": {
    "kind": "page",
    "url": "$.url",
    "title": "$.title"
  },
  "employee_rows": {
    "kind": "extracted",
    "message": "Read ${$.length} employee rows"
  }
}

Scan rule

Only the top of the script is scanned. Blank lines, shebangs, and comments before the block are allowed. Real code before the block stops the scan.

Selector rule

$.url pulls a field from the emitted value. $.length reads the length of a list, dict, or string. Template strings substitute selector results.

Fallback rule

If a labeled output has no matching spec, the full output is still preserved and the UI gets a generic Recorded label summary.

Model-visible channel

outputs[] contains the full emitted value. This is where large page info, extracted rows, selected DOM state, and API results remain available for the next model turn. 

{
  "label": "page_info",
  "value": {
    "url": "https://news.ycombinator.com/",
    "title": "Hacker News",
    "...": "full data"
  },
  "summary": { "...": "derived compact record" }
}

Frontend-visible channel

summary[] is the preferred transcript surface. The TUI can render this deterministically without dumping the raw object. 

{
  "kind": "page",
  "url": "https://news.ycombinator.com/",
  "title": "Hacker News",
  "output_label": "page_info"
}
Question Answer
Does the comment run as Python? No. It is parsed from the source string before Python executes the user code.
How does the agent know labels? The agent writes the labels in emit_output(..., label="..."). The summary block repeats those labels as keys.
Can the block be added after the script starts? No. The runtime parses the submitted source once before execution. Authoring can happen code-first, but the final submitted code needs the block at the top.
Can information be lost? The display summary can be compact, but the full emitted value stays in outputs[] for the model.

Mental model: # browser_summary is a label-to-display-spec lookup table. emit_output is still the single structured data emission point.
