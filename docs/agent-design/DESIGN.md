## Build order

- Wave 0 (serial, gating): WP-A0 — land the compiling frozen scaffold (lib.rs module tree + error/config + testkit trait fakes with unimplemented!() bodies). Every later WP branches from this single commit so module paths/signatures are frozen. Merge first, alone.
- Wave 1 (fully parallel, 6 isolated worktrees — all PURE, no async, disjoint files): WP-A1 decision::loop_decision/retry; WP-A2 tools trait seam + decision::tool_decision; WP-A3 context accounting/assembly/normalize/image; WP-A5 events pure mapper; WP-A6 session pure reducer/rollback; (WP-A4 starts after A3 since inject reuses assembly). Merge one at a time; each only touches its own files so merges are trivial.
- Wave 1b: WP-A4 context inject (depends on A3 merged).
- Wave 2 (parallel, async wrappers over the now-merged pure cores — disjoint files): WP-B1 tools orchestrator (needs A2); WP-B2 context manager wrapper (needs A3+A4); WP-B3 events store_sink (needs A5); WP-B4 session async lifecycle (needs A6); WP-B5 turn sampling driver (needs A1+A5). Five worktrees, merge one at a time.
- Wave 3 (parallel where possible): WP-C1 turn dispatch (needs B1+A1); then WP-C2 turn loop driver (needs A1+B5+C1+B2) — this is the integration spine; finally WP-C3 task driver/abort/lifecycle (needs C2+B4+B3). C1 parallel with the tail of Wave 2; C2 then C3 are effectively serial (C3 wraps C2).
- Rationale: the whole PURE decision surface (Wave 1) is independently unit-testable against codex parity tables before ANY async exists; Wave 2 async wrappers each delegate to a merged pure core, so they cannot diverge in behavior; Wave 3 is thin integration. Max parallelism is 6 (Wave 1) and 5 (Wave 2).

## Parity test plan

Cross-cutting principle: mirror browser-use-llm's tool_runtime ScriptedTurnSource (read at /tmp/llm_toolrt.txt; ScriptedTurnSource + reduce_turn) — every async surface is driven by scripted, deterministic fakes from testkit so no network/disk is touched, and every behavior branch is proven on the PURE fn it delegates to.

WP-A1 (loop decision): exhaustive truth tables vs codex turn.rs:168-355. needs_follow_up over {model_nfu x has_pending_input}; classify_loop_step over the 8-cell cube {model_needs_follow_up x has_pending_input x token_limit_reached} -> {Compact/Continue/Complete}; can_drain_after_compact; initial_can_drain; retry_decision over {retries<max, retries==max+can_switch, retries==max+!can_switch, !retryable} -> {Backoff/SwitchTransport/Fail}; backoff_ms determinism.

WP-A2 (tool decisions): table-driven tests reproducing codex sandboxing_tests.rs:23-94 exactly — default_exec_approval_requirement over {Never,OnFailure,OnRequest+Restricted,OnRequest+Unrestricted,UnlessTrusted,Granular-disabled}; sandbox_override_for_first_attempt over {Skip-bypass, denied-read, escalated-perms, plain}; should_bypass_approval / wants_no_sandbox_approval default-impl matrices; plan_attempts decision table (initial approval needed?, initial sandbox None vs Restricted, on-denial Return vs RetryNone{reapprove}); map_decision over every ReviewDecision -> Ok/Err(Rejected).

WP-A3 (accounting/assembly): hand-computed codex values — estimate_reasoning_length(1000)=100; estimate_encrypted_function_output_length(16)=9; approx_tokens_from_byte_count_i64(7373)=1844; RESIZED replacement=7373. PORT codex context_manager/history_tests.rs fixtures verbatim (they build Item vecs and assert for_prompt/normalize/token totals). total_token_usage with scripted TokenUsageInfo{last.total=1000} + locally-added tail; the +non_last_reasoning branch when server_reasoning_included=false. Provider-Usage mapping: build browser-use-llm Usage (event.rs Usage{input,output,reasoning,cached, computed_total excludes cached — confirmed /tmp/event_rs.txt) and assert TokenUsage.total=input+output+reasoning. Property test: for_prompt pairs every call with exactly one output and is idempotent.

WP-A4 (inject): construct named context messages; assert move_workspace_context_before_first_user puts permissions then workspace (environment section LAST) before first plain user; is_contextual_message keys off `name`; build_settings_update_items emits fixed section order and None when include_* off/unchanged; None baseline => full reinjection.

WP-A5 (events mapper): scripted Vec<LlmEvent> following the confirmed ordering (StepStart -> Text/Reasoning/ToolInput deltas -> StepFinish -> Finish, plus ProviderError — confirmed /tmp/event_rs.txt) -> assert exact Vec<PendingEvent> (type strings + payload JSON) vs golden fixtures. usage_to_model_usage (cached-subset, total==0 -> computed_total). token_count_payload running-total preservation when usage absent. session_done_payload precedence (result_file pointer vs inline; normalize_result_text 24-char dedup). STRONGEST guard: feed produced events (stamped into EventRecord) through the REAL browser-use-protocol projections (transcript_from_events/result_from_events/activity_from_events) and assert TUI-visible output matches legacy — those projections are the actual consumers, so matching == cutover-safe.

WP-A6 (session reducer): PORT the legacy crate's OWN golden vectors at browser-use-core/src/lib.rs:26916-29761 (dozens of provider_messages_from_events assertions) verbatim — cover model.delta(assistant_delta_to_append) vs model.message dedup, tool.call/tool.output pairing + emitted_tool_messages dedup, tool.failed/finished synthetic results, session.cancelled/done/failed suppress_terminal_tail, workspace/developer context-by-before_seq, compaction-checkpoint resume, agent.context fork seed. Rollback table tests (rollback_filtered_events_after, num_turns||turns||n fallback, before_seq contextual stripping). is_persistable_event allowlist vs policy.rs mapping.

WP-B1 (orchestrator): ScriptedApprover (queued Vec<ReviewDecision>, records ApprovalRequest payloads) + ScriptedToolRuntime (queued Vec<Result<Out,ToolError>>, e.g. Err(Sandboxed) then Ok) — assert number of run() attempts, sandbox of each (None vs Restricted), whether re-approval requested (+ retry_reason / call_id:retry id), final result vs ToolError. ApprovalStore: pre-seed keys=ApprovedForSession, assert fetch NOT called. Stub path: ToolOrchestrator::stub() over a real ShellRuntime runs unsandboxed with no prompt (proves a tool implemented once works under sandbox=None today).

WP-B2 (context manager): in-memory/no-op StoreSink; assert it NEVER reads back (write-sink-only invariant — EventSource half panics on read).

WP-B3 (store_sink): temp SQLite Store::open_with_notifier(tmp, tx); emit a sequence; assert append order via events_for_session AND that StoreNotification::EventsChanged{seq} fired on rx (no polling).

WP-B4 (session async): in-memory FakeEventSink+EventSource (Vec<EventRecord> behind Mutex): resume(reduce(appended)) == live history; the runtime write path NEVER calls events_for_session (read half panics). Coordination test with tokio broadcast fake: waiter wakes on append within one notify (a sink that NEVER sleeps proves the 50ms poll is gone). Status test: append session.done -> resumed status()==Done (status_for_event parity).

WP-B5 (sampling): ScriptedSamplingDriver returns pre-baked SamplingOutcomes; a fake ModelClient streams scripted LlmEvents incl. ProviderError to exercise retry_decision -> Backoff/SwitchTransport (use tokio::time::pause/advance for deterministic delays).

WP-C1 (dispatch): feed a fixed call list whose futures resolve OUT OF ORDER (oneshot ordering) and assert outputs recorded in MODEL order (FuturesOrdered parity); assert a Serial(write) call blocks concurrent Parallel(read) calls via a shared counter; cancel-then-drain path observes already-started calls before TurnAborted.

WP-C2 (turn loop): ScriptedSamplingDriver returning a sequence (e.g. [nfu, nfu, complete]) + InMemoryTurnState (Vec history + VecDeque pending steer + scriptable TokenStatus) + RecordingTurnSink: assert exactly N sampling requests; steer drained at the right iterations (can_drain gating); compaction triggers exactly when token_limit_reached && needs_follow_up; TurnStarted/TurnComplete emitted once.

WP-C3 (task driver): tokio::time::pause/advance to prove hard abort fires after ~100ms against a task that ignores cancellation; assert TurnAborted emitted with interrupted-turn marker recorded BEFORE the event when reason==Interrupted; spawn_task aborts the prior active turn (Replaced) before starting; only one ActiveTurn (debug_assert).

CI gate: a single `cargo test -p browser-use-agent` runs all of the above with zero network/disk; the protocol-projection cross-check (WP-A5) and the ported legacy golden vectors (WP-A6) + ported codex history fixtures (WP-A3) are the three byte-for-byte parity anchors.

## Risks

- Canonical Item type unresolved: estimate_response_item_model_visible_bytes serializes the item, so ANY serde drift breaks every byte/token estimate. browser-use-protocol does NOT yet expose a ResponseItem-equivalent enum (grep returned nothing). The frozen interface aliases context::Item = serde_json::Value (legacy provider-message currency, which is what browser-use-core actually uses and what session reconstruction returns). DECISION NEEDED before WP-A3/A6: keep Value (byte-parity with legacy) vs introduce a typed enum. Recommend Value now (matches legacy provider_messages_from_events) and add a Value<->browser-use-llm::Message adapter in ContextManager::lower_to_messages.
- tools/parallel.rs RwLock mechanics could not be read line-by-line (Read returned empty past turn.rs:1518 and for parallel.rs in the original analysis). The per-tool parallel-safety predicate is grounded in the ToolCallRuntime import + supports_parallel_tool_calls (turn.rs:872) but the exact gate must be reconciled when WP-C1 is built; the ToolRuntime::parallel_safe() default is conservatively false.
- SamplingDriver vs ToolDispatcher boundary: codex FUSES the model stream and tool dispatch inside try_run_sampling_request (turn.rs:1689) for shared cancellation/drain semantics. The proposed design SPLITS them (cleaner, but risks divergence in drain_in_flight timing). The split must be validated against the cancel-then-drain path in WP-C1/C2; if parity fails, fuse SamplingDriver+ToolDispatcher into one trait.
- Confirmed: legacy browser-use-core has NO max_turns/max_iterations cap (grep across the crate found none in this session — the probe output capture was intermittent but consistent with the original analysis). The 'drop max_turns=80' requirement is therefore likely a NO-OP, not an active removal; verify once more at WP-C2 implementation time so the unbounded-loop claim is genuinely a non-regression.
- Network-approval lifecycle (Immediate/Deferred, NetworkPolicyDecision::host) is a placeholder in codex grounding; the orchestrator (WP-B1) depends on begin/finish_network_approval being finalized by a sibling network-approval subsystem. For sandbox=None initial scope the NoneSandboxProvider never denies, so network approval is inert — but the trait surface must not bake in assumptions that block the real impl.
- server_reasoning_included flag source: total_token_usage selects a branch on it (history.rs:303-332). It must be surfaced from browser-use-llm route/Protocol capabilities; currently carried on AgentConfig as a plain bool. If the real flag is per-model/per-protocol, AgentConfig must be refreshed per turn or the branch will be wrong after a model switch.
- Output-capture flakiness in this analysis environment (bash stdout intermittently dropped; worked around via tmp-file + Read). Some deeper codex bodies (try_run_sampling_request 1689, drain_in_flight 1655, plan-mode streaming split 1380-1602) were located by grep but not fully byte-read; the event-stream ordering parity (emit ItemStarted/Done vs record-to-history vs needs_follow_up accumulation) needs a confirming read during WP-C1/C2 for byte-level parity.
- StoreSink seq-pairing throughput: tool.image/artifact.created need the assigned event.seq to call record_artifact(Some(seq)), forcing emit_and_record to AWAIT the seq, while the common path stays fire-and-forget. The writer-task channel design (StoreOp enum) must keep these two paths from head-of-line-blocking each other; validate in WP-B3.
- Fork seed payload shape (agent.context with fork_response_items + history_mode) must be byte-exact so the child rebuilds the helper-session-identity system message identically (lib.rs:9188-9230) and is independently resumable. This straddles WP-A6 (reducer consumes it) and WP-B4 (lifecycle emits it); a mismatch silently corrupts forked children. Add a round-trip fork->resume golden test.
